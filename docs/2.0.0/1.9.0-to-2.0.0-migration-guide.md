# 1.9.0 → 2.0.0 Migration Guide (**DRAFT**)

> ⚠️ **DRAFT** — This guide is a 2.0.0 cycle planning artifact published in 1.9.0-RC19. The final
> migration guide ships with 2.0.0 GA. Feedback welcome via GitHub issues. Subject to change during
> the 2.0.0 cycle.

---

## Audience

You are running netty-spring **1.9.0** today and want to upgrade to **2.0.0** so you can run on
**Spring Boot 3.x**. This guide tells you what changes for *you* — the application developer — and
walks you through the bump in concrete steps.

If you have **no plans to move off Spring Boot 2.7.18**, you do not need to upgrade. 1.9.0 is the
end-of-line GA release for the Boot 2.7 era and continues to receive security backports on its
own track.

---

## TL;DR

The migration is smaller than you probably expect.

- netty-spring runs on **raw Netty**, not the Servlet API. The `javax.servlet` → `jakarta.servlet`
  sweep that dominates Boot 3.x migrations **does not apply to netty-spring itself**.
- Across the entire reactor, exactly **one** import moves from `javax.*` to `jakarta.*` —
  `javax.activation.MimetypesFileTypeMap` in `ServiceHandlerUtil`. (Source:
  `docs/2.0.0/boot3-compatibility-matrix.md`.) All `javax.crypto.*` imports are JDK APIs and stay.
- Existing `application.yml` keeps working. New keys (sharded pub/sub, Observation tracing) are
  **opt-in**.

The work is almost entirely in *your* application code, not in netty-spring.

## Breaking changes (expected)

### Spring Boot 2.7.18 → 3.2.x

You bump the Boot version. This requires:

- **JDK 17 minimum.** netty-spring 1.9.0 already requires JDK 17, so if you are on 1.9.0 today,
  you are already good. If you are still on JDK 11, upgrade the JDK first as a 1.9.0 task — not
  bundled with the 2.0.0 upgrade.
- **Spring Framework 5.3.x → 6.x.** The dominant API surface netty-spring uses
  (`ApplicationContextRunner`, `@ConditionalOnBean/Class/Property/MissingBean`, `@AutoConfigureAfter`,
  `EnableConfigurationProperties`) is stable across this boundary. No source change required in
  your auto-configurations.

### `javax.*` → `jakarta.*` rename

For **your** application code, the standard rule applies: if you use the Servlet API, JPA,
Bean Validation, `@PostConstruct`/`@PreDestroy`, or `javax.activation`, mass-rename to `jakarta.*`.
A typical one-shot sed handles it:

```bash
# Run from your project root. Mac/BSD sed: replace -i with `-i ''`.
find src -name '*.java' -exec sed -i 's/javax\.servlet/jakarta.servlet/g' {} +
find src -name '*.java' -exec sed -i 's/javax\.annotation\.PostConstruct/jakarta.annotation.PostConstruct/g' {} +
find src -name '*.java' -exec sed -i 's/javax\.annotation\.PreDestroy/jakarta.annotation.PreDestroy/g' {} +
```

Most pure netty-spring users will find zero matches — the framework's whole point is that you
write Netty pipelines, not Servlets.

### Auto-config registration file (transparent if you only consume starters)

Internally, 2.0.0 moves starter modules from `META-INF/spring.factories` to
`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (the format Boot
3 mandates). **You do not have to do anything for this** unless you wrote your own auto-config
module that depended on netty-spring's `spring.factories` shape.

## Non-breaking additive features

### Sharded pub/sub (new in 2.0.0)

A new `RedisShardedPubSubBroker` will partition cross-node broadcast across Redis Cluster shards
using `SSUBSCRIBE` / `SPUBLISH` (Lettuce 6.2+, Redis 7.0+). The proven RC7 `RedisClusterModePubSubBroker`
remains the default — sharded mode is **opt-in**. See `docs/2.0.0/sharded-pubsub-feasibility.md`
for the design study.

### Micrometer Observation API trace continuation

The RC6-deferred Observation-based cross-node trace continuation lands in 2.0.0 once Boot 3.x
brings Micrometer 1.10+. The existing `ClusterTraceContext` SPI (MDC-only) stays in place; an
`ObservationClusterTraceContext` implementation becomes available behind a config switch. **Existing
deployments need no change** unless they want to flip the switch.

## Step-by-step upgrade

1. **Bump the netty-spring version** in your POM:

   ```xml
   <dependency>
     <groupId>io.github.berrywang1996</groupId>
     <artifactId>netty-websocket-cluster-spring-boot-starter</artifactId>
     <version>2.0.0</version>     <!-- was 1.9.0 -->
   </dependency>
   ```

2. **Verify JDK ≥ 17.** No change for users already on 1.9.0:

   ```bash
   java -version       # expect 17.x or newer
   mvn -v              # confirm Maven's chosen JDK
   ```

3. **Bump Spring Boot** in your parent POM (or BOM):

   ```xml
   <parent>
     <groupId>org.springframework.boot</groupId>
     <artifactId>spring-boot-starter-parent</artifactId>
     <version>3.2.x</version>   <!-- was 2.7.18 -->
   </parent>
   ```

   Pick whichever Boot 3 LTS is OSS-supported when you upgrade.

4. **Mass-rename `javax.*` to `jakarta.*`** in your application code (Servlet, JPA, Bean Validation,
   `@PostConstruct`/`@PreDestroy`, `javax.activation`). See the sed snippets above. Pure netty-spring
   apps will commonly find zero matches.

5. **Find stragglers**:

   ```bash
   mvn dependency:tree | grep -i javax
   grep -rn "^import javax\." src --include='*.java'
   ```

   `javax.crypto.*` matches are JDK APIs — leave them alone. Everything else is a candidate.

6. **Run the tests**:

   ```bash
   mvn clean test
   ```

   Green build = you are done.

## New feature opt-ins

### Sharded pub/sub

Already using Redis Cluster (`cluster-nodes` set)? Flip one knob:

```yaml
server:
  netty:
    websocket:
      cluster:
        redis:
          cluster-nodes: redis-0:6379,redis-1:6379,redis-2:6379
          pubsub-mode: sharded     # NEW in 2.0.0; requires Redis 7.0+
```

Requires **Lettuce ≥ 6.5.5** (handled by Boot 3.2+'s BOM) and **Redis ≥ 7.0**. Standalone /
Sentinel deployments cannot use sharded mode — startup fails fast.

### Observation API trace continuation

Add `micrometer-tracing` to your application and the RC6 MDC-only `ClusterTraceContext` becomes a
no-op while the new Observation-based bridge takes over. Brave / OpenTelemetry collectors then pick
up cross-node spans without further configuration. The custom `ClusterTraceContext` SPI override
path stays in place if you need it.

## Configuration compatibility

**All 1.9.0 `application.yml` configuration continues to work in 2.0.0.** Every new key (sharded
pub/sub, Observation toggle, any others added during the cycle) is additive with a default that
preserves 1.9.0 behavior. If we are ever forced to change a default, this guide and the 2.0.0
release notes will call it out explicitly.
