## What worked

- **One RC = one theme.** Every RC mapped to a single scope line in `cluster-design.md` (broker /
  registry / metrics / E2E / NATS / audit). Readers could skim a one-paragraph release-notes
  section and know exactly which subsystem changed.
- **Adversarial multi-agent audit, twice.** RC11 (pre-GA, 33 raw → 24 confirmed → 15 shipped) and
  RC17 (GA-readiness, 10 dimensions → 3 must-fix) both surfaced **real, mechanical issues** that
  per-PR review missed: missing Spring metadata entries (silent IDE autocomplete degradation),
  module-wide missing Apache 2.0 license headers (32 cluster + 7 starter test files), URI base64url
  encoding gap in registry keys. The pattern paid for itself twice.
- **Backlog discipline.** Every nice-to-have got an ID (L / P / Q / R / S / T series), a filed entry
  in `docs/pre-ga-audit-backlog.md`, and was either shipped or explicitly refuted in writing. The
  backlog **emptied** at RC16. Q4 is the textbook example: the reviewer flagged a `DedupRing`
  capacity boundary as a bug; RC14 brainstorm re-read the code and refuted it with citations to
  the `LinkedHashMap` constructor semantics. The refutation is in the backlog file.
- **Forward-looking docs deliberately separated.** RC19's three 2.0.0 prep docs sit under
  `docs/2.0.0/` with explicit "this is NOT a 1.9.0 commitment" banners. Clean signal for any future
  reader of shipped-vs-planned.
- **Behavior preservation.** Single-node mode (`cluster.enable=false`, the default) remained
  byte-level identical to 1.7.x / 1.8.0 across all 19 RCs. The only intentional default change was
  `redis-loss-grace-period-ms=5000` (RC1) — and even that has a `0` escape hatch.

## What didn't quite work

- **Polish-RC inflation.** RC14 (6 items), RC15 (8 items), RC18 (4 items) could plausibly have been
  bundled into one or two larger RCs. The "ship every dimension I find" instinct led to small RCs
  in the polish phase. Each one still carried full release-notes + spec + plan + finish-branch
  overhead.
- **Reviewer false positives reaching the spec-edit phase.** Q4 (`DedupRing` capacity) made it
  into the RC13 review backlog before being refuted in RC14 by a careful re-read. The cost is
  small (one refutation paragraph) but ideally implementer-side reading would catch this before
  it escalates one RC.
- **Audit dimension overlap.** RC17's 10-dimension review had D7 (configuration completeness) and
  D9 (release engineering / Apache 2.0 headers) producing file-level overlap that was only
  disambiguated at synthesis. Rename dimensions or have synthesis explicitly de-duplicate at the
  auditor handoff.
- **A 1.8.0+ silent JSON bug should have been caught well before RC8.** The multi-node Docker demo
  smoke (RC8) discovered that `DefaultMessagePayloadCodec` was serializing cross-node JSON broadcasts
  via `toString()` (Map → `{a=b}`, not JSON), so cross-node JSON delivery was unparseable. **This bug
  shipped in 1.8.0 and survived RC1–RC7** because local delivery uses Jackson directly and unit tests
  covered only the local path. Lesson: **test fixtures must exercise the cross-process serialization
  path** for any payload codec, not assume local-path parity. RC8 backfilled the regression test
  (`DefaultMessagePayloadCodecTest`, 6 cases) — but the gap had been open ~6 months across two cycles.

## Lessons for 2.0.0

- **Set an RC count budget upfront.** The 1.9.x tail (RC14 / RC15 / RC18 / RC19) drifted into
  individually-small RCs. A 2.0.0 plan that declares "Boot 3.x cycle = 5–8 RCs, decided at the
  start" is healthier. RC19's own cycle-shape estimate (5–7 RCs) is consistent with this.
- **The Boot 3.x jakarta sweep is *not* the bulk of the work.** RC19 Track M measured the entire
  reactor and found **exactly one `javax.activation` file** affected (`ServiceHandlerUtil`). Zero
  `javax.servlet`, zero `@PostConstruct` / `@PreDestroy`, zero `javax.persistence`. netty-spring
  runs on raw Netty and never depended on the Servlet API — the jakarta sweep is a one-import
  change here. **The real bulk is** Lettuce 6.5.5 (auto-resubscribe correctness for sharded
  pub/sub) + Micrometer Observation API (replaces RC6's MDC-only trace propagation).
- **Sharded pub/sub is the only real architectural unlock.** RC19 Track P quantified ~83% per-shard
  fan-out reduction at k=6 masters via `SSUBSCRIBE`/`SPUBLISH`. Everything else in the 2.0.0 plan
  is mechanical migration. Allocate review effort accordingly.
- **The audit-driven QA pattern is reusable.** Run the same pre-GA + GA-readiness audit pair in
  2.0.0. **Calibrate audit budget to RC size** — a 6-RC Boot 3.x cycle doesn't need 10 dimensions
  per pass; 6–8 suffices.

## The audit-driven QA pattern (meta)

The shape of the RC11 pre-GA audit and the RC17 GA-readiness audit is worth replicating verbatim:

| Step | What |
|---|---|
| 1 | Multi-lens parallel auditors (8–10 dimensions, sized to scope). Each auditor reads ~10 files independently. |
| 2 | Adversarial skeptic verifier per HIGH / MEDIUM finding — refuted items dropped before synthesis. |
| 3 | Synthesis agent emits machine-readable verdict: `rc-ready` / `must-fix list` / `nice-to-have backlog`. |
| 4 | Two-stage cadence: pre-GA (mid-cycle) + GA-readiness (end-of-cycle). Both caught real issues in 1.9.x. |

Cost: ~1.1M subagent tokens, ~8 min wall-clock per run (RC17 measurement). Output: JSON report +
human-readable narrative, both check-in artifacts. Key properties: parallel dimensions make the
audit fast at high coverage, and the adversarial skeptic step drives the false-positive rate near
zero — findings that survive synthesis are nearly always real and actionable.
