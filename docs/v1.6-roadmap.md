## Phase 1: 广播性能优化 (核心瓶颈修复)

> 预估工作量: 3-4 周

### 1.1 零拷贝广播序列化 (最高优先级)

**问题**: `topicMessage()` 对每个 session 调用 `message.responseMsg()`，触发 N 次 Jackson 序列化 + N 次 ByteBuf 分配。

**方案**: 序列化一次，通过 `ByteBuf.retainedDuplicate()` 共享 payload。

```java
// 新的广播路径
public void topicMessage(String uri, AbstractMessage message) {
    // Step 1: 序列化一次 (调用者线程)
    ByteBuf payload = serializeOnce(message); // PooledByteBufAllocator.DEFAULT.directBuffer()
    try {
        Map<EventLoop, List<MessageSession>> grouped = groupByEventLoop(sessions);
        for (Map.Entry<EventLoop, List<MessageSession>> entry : grouped.entrySet()) {
            ByteBuf elPayload = payload.retainedDuplicate();
            entry.getKey().execute(() -> {
                try {
                    for (MessageSession s : entry.getValue()) {
                        if (!s.getChannelHandlerContext().channel().isWritable()) continue;
                        // retainedDuplicate() 共享底层内存，独立引用计数
                        TextWebSocketFrame frame = new TextWebSocketFrame(elPayload.retainedDuplicate());
                        s.getChannelHandlerContext().write(frame); // write without flush
                    }
                    // 批量 flush: 合并 syscall
                    for (MessageSession s : entry.getValue()) {
                        s.getChannelHandlerContext().flush();
                    }
                } finally {
                    elPayload.release();
                }
            });
        }
    } finally {
        payload.release();
    }
}
```

**关键设计决策**:
- 使用 direct buffer (off-heap)：避免每次 write 时 heap→direct 拷贝
- `retainedDuplicate()` 而非 `slice()`：自动管理引用计数，线程安全
- 按 EventLoop 分组：减少跨线程提交，从 N 个任务降为 EventLoop 数量 (~2*cores)
- `write()` + 批量 `flush()`：将 N 次 syscall 合并为 EventLoop 数量次

**性能预期**: 500 人广播从 500 次序列化+syscall 降为 1 次序列化 + ~4 次 syscall (2C 环境)，**预期 10-20x 吞吐提升**。

### 1.2 AbstractMessage API 扩展

```java
public abstract class AbstractMessage<T> {
    // 现有: 每次创建新 Frame (保持向后兼容)
    public abstract T responseMsg();

    // 新增: 返回可共享的 ByteBuf payload
    public ByteBuf serializePayload(ByteBufAllocator allocator) {
        // 默认实现: 回退到 responseMsg() 路径
        T frame = responseMsg();
        if (frame instanceof WebSocketFrame) {
            return ((WebSocketFrame) frame).content().retain();
        }
        throw new UnsupportedOperationException();
    }
}
```

**变更文件**: `AbstractMessage.java`, `JsonMessage.java`, `TextMessage.java`, `BinaryMessage.java`

### 1.3 FlushConsolidationHandler 集成

在 `NettyChannelInitializer.initChannel()` pipeline 中添加：

```java
p.addLast(new FlushConsolidationHandler(256, true));
```

参数 `consolidateWhenNoReadInProgress=true` (Netty 4.1.46+) 确保广播场景(非 channelRead 触发)也能合并 flush。

**配置属性**: `server.netty.websocket.flush-consolidation-threshold=256`

### 1.4 Per-Session 背压替代全局 DROP/CALLER_RUNS

**现状**: 线程池满时全局策略——一个慢 session 导致所有 session 丢消息。

**方案**: 
- 使用 `channel.isWritable()` + `WriteBufferWaterMark` 做 per-session 背压
- 新增 per-session 有界溢出队列 (可配置大小)
- `channelWritabilityChanged` 事件触发排空

```yaml
server.netty.websocket:
  write-buffer-low-water-mark: 32768    # 32KB
  write-buffer-high-water-mark: 65536   # 64KB
  per-session-overflow-queue-size: 64   # 每 session 最多缓存 64 条消息
```

### 1.5 线程模型迁移

**弃用** sender ThreadPoolExecutor，改为直接提交到 EventLoop：

| 维度 | 旧模型 (ThreadPoolExecutor) | 新模型 (EventLoop Direct) |
|---|---|---|
| 任务数/次广播 | N (每 session 一个) | EventLoop 数量 (~2*cores) |
| 上下文切换 | ThreadPool → EventLoop 唤醒 | 0 (已在 EventLoop 内) |
| 背压粒度 | 全局 (线程池饱和) | Per-session (channel.isWritable) |
| 配置复杂度 | 需调优 core/max/queue | 无需配置 |

保留 `broadcastMode=THREAD_POOL_LEGACY` 配置项用于向后兼容。

## Phase 3: 集群支持 (分布式扩展)

> 预估工作量: 6-8 周 | 新增模块: `netty-spring-cluster`

### 3.1 架构设计

```
                    ┌─── Load Balancer (Nginx/K8s Ingress) ───┐
                    │         sticky session by IP/cookie       │
                    ▼              ▼              ▼              ▼
              ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
              │  Node-1  │  │  Node-2  │  │  Node-3  │  │  Node-N  │
              │ netty-   │  │ netty-   │  │ netty-   │  │ netty-   │
              │ spring   │  │ spring   │  │ spring   │  │ spring   │
              │ instance │  │ instance │  │ instance │  │ instance │
              └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
                   │              │              │              │
                   ▼              ▼              ▼              ▼
              ┌──────────────────────────────────────────────────────┐
              │                    Redis Cluster                      │
              │  ┌──────────┐ ┌──────────┐ ┌──────────────────────┐ │
              │  │ Pub/Sub  │ │ Session  │ │  Streams (可选)      │ │
              │  │ Channels │ │ Registry │ │  持久化消息/离线消息 │ │
              │  └──────────┘ └──────────┘ └──────────────────────┘ │
              └──────────────────────────────────────────────────────┘
```

### 3.2 集群发现与成员管理

**推荐方案: Redis Heartbeat + 事件通道** (最小外部依赖)

```
Redis Key 设计:
  netty:cluster:nodes              → Hash { nodeId → JSON(host, port, startTime, lastHeartbeat, sessionCount) }
  netty:cluster:heartbeat:{nodeId} → String with TTL 30s (心跳 key，过期即判定故障)
  netty:cluster:control            → Pub/Sub channel (节点加入/离开/广播命令)
```

**故障检测**:
- 每个节点每 10s 更新 `heartbeat:{nodeId}` 的 TTL
- 使用 Redis Keyspace Notification 监听 key 过期事件
- TTL 30s → 最大故障检测延迟 30s
- 对于需要更快检测的场景 (< 5s)，可选切换到 ScaleCube SWIM 协议

**节点生命周期**:
```
JOINING → ACTIVE → DRAINING → LEFT
```

### 3.3 分布式 Session 管理

```
Redis Key 设计:
  netty:session:{uri}:{sessionId} → Hash { nodeId, userId, connectedAt, metadata }
  netty:node:{nodeId}:sessions    → Set { sessionId1, sessionId2, ... }
  netty:user:{userId}:sessions    → Set { "uri:sessionId@nodeId", ... }  (可选，用于按用户查询)
```

**会话路由**: 私聊消息 A→B：
1. 查 `netty:session:{uri}:{targetSessionId}` 获取 `targetNodeId`
2. 如果 `targetNodeId == 本节点` → 本地发送
3. 否则 → 发布到 `netty:cluster:unicast:{targetNodeId}` channel

### 3.4 跨节点消息路由

#### 广播 (Pub/Sub)

```
每个聊天室 URI = 一个 Redis Pub/Sub channel:
  netty:broadcast:{uri}

节点只订阅自己有活跃 session 的 URI channel。
```

**流程**:
1. Node-1 上用户发广播消息
2. Node-1 **本地扇出** (写入本节点的所有 session)
3. Node-1 **发布到 Redis** `netty:broadcast:{uri}`
4. Node-2, Node-3 **收到订阅消息** → 各自本地扇出

**优化**: 只有在 URI 有跨节点 session 时才发布到 Redis。纯本地 URI 直接走内存路径。

#### 私聊 (点对点)

```
netty:cluster:unicast:{targetNodeId}  → Pub/Sub channel (每节点一个)
```

低延迟: Redis Pub/Sub RTT 通常 < 1ms (同机房)。

#### 可靠消息 (可选, 使用 Redis Streams)

```
netty:stream:{uri}  → Redis Stream (带 consumer group)
```

用于需要 at-least-once 投递保证的场景 (离线消息、重要通知)。

### 3.5 弹性扩缩容

**Scale-out (扩容)**:
1. 新节点启动 → 注册到 `netty:cluster:nodes` → 发布 JOIN 事件
2. Load Balancer 将新连接路由到新节点
3. 无需迁移现有连接 — 自然平衡

**Scale-in (缩容)**:
1. 节点收到 DRAIN 信号 → 设置状态为 DRAINING
2. 停止接受新连接 (从 LB 摘除)
3. 向所有 session 发送 `CloseFrame(1001, "Server going away")`
4. 客户端重连到其他节点 (需客户端实现重连逻辑)
5. 等待所有 session 关闭或超时 → 从 Redis 注销 → 退出

**Kubernetes HPA 集成**:
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  metrics:
  - type: Pods
    pods:
      metric:
        name: netty_ws_connections_active  # 来自 Prometheus
      target:
        type: AverageValue
        averageValue: "5000"  # 每 Pod 目标连接数
```

### 3.6 自恢复机制

**节点故障恢复**:
1. 故障节点的 heartbeat key TTL 过期
2. 其他节点收到 Keyspace Notification
3. 触发清理: 从 `netty:session` 中删除故障节点的所有 session 记录
4. 发布 `NODE_LEFT` 事件到 control channel
5. 客户端感知到连接断开 → 自动重连到其他健康节点

**脑裂防护**:
- Redis 作为 single source of truth，天然避免脑裂
- 如果节点与 Redis 断连 > 配置超时 → 主动关闭所有本地 session (保守策略)

### 3.7 API 设计

```java
// 集群模式对应用代码完全透明
// MessageSender 接口不变，ClusterMessageSender 自动路由
@Controller
public class ChatController {
    private final MessageSender messageSender; // 自动注入集群版本

    @MessageMapping(value = "/ws/chat", messageType = MessageType.TEXT_MESSAGE)
    public void onMessage(ChatMessage msg, MessageSession session) {
        // broadcastJson 自动: 本地扇出 + Redis Pub/Sub 发布
        messageSender.broadcastJson("/ws/chat", response);

        // sendJsonToSession 自动: 本地 or Redis 路由
        messageSender.sendJsonToSession("/ws/chat", pm, targetSessionId);
    }
}
```

**自动配置**:
```yaml
# application.yml
server.netty.cluster:
  enabled: true                          # 启用集群模式
  redis-uri: redis://localhost:6379      # Redis 地址 (或使用 Spring Data Redis 配置)
  node-id: ${HOSTNAME:auto}              # 节点 ID (默认自动生成)
  heartbeat-interval: 10s               # 心跳间隔
  heartbeat-timeout: 30s                # 故障检测超时
  drain-timeout: 60s                    # 缩容排空超时
```

```yaml
# 不配置 cluster.enabled 或设为 false → 单机模式 (完全兼容 v1.5.x)
```
