## 2. Deliverables

### 2.1 `docs/2.0.0/sharded-pubsub-feasibility.md`

Design study for `RedisShardedPubSubBroker` — the natural successor to RC7's `RedisClusterModePubSubBroker`
that uses Lettuce 6.2+ SSUBSCRIBE/SPUBLISH instead of regular cluster pub/sub.

**Required sections:**
- **Motivation** — the M·(f·N-1) fan-out wall from cluster-design.md §Capacity (currently bypassed only
  by going horizontal / running fewer URIs / accepting back-pressure)
- **Why deferred from 1.9.0 (RC7)** — Boot 2.7.18 manages Lettuce 6.1.10; sharded pub/sub needs 6.2+;
  Boot 3.x manages 6.2+. Sharded is a 2.0.0-only feature.
- **API surface** — Lettuce 6.2's `RedisClusterPubSubAsyncCommands.ssubscribe(...)` /
  `.spublish(channel, message)`; the SSUBSCRIBE-only consumer-keyspace partitioning by hash-slot;
  Redis 7.0+ requirement
- **Fan-out reduction model** — quantitative: original M·(f·N-1) → sharded approx M·(f·N-1)/k where k is
  the cluster master count. Table for f=1, k=3/6/12 cluster masters, N=100/500/1000 sessions.
- **Auto-config matrix entry** — new bean `RedisShardedPubSubBroker` activated by
  `cluster-nodes` (non-empty) AND `pubsub-mode=sharded` (new opt-in property) AND Lettuce 6.2+ on
  classpath. Default `pubsub-mode=regular` for RC7-parity.
- **Mutual-exclusion** — sharded broker doesn't change session-registry layout. The existing
  `RedisClusterModeSessionRegistry` (RC7) is reused. Sharded broker is purely the cross-node fan-out.
- **Test strategy** — 6-master Testcontainers Redis Cluster (vs RC7's single-node-cluster), shard
  distribution verification, fan-out reduction measured vs regular cluster pub/sub baseline.
- **Migration impact** — opt-in; existing `cluster-nodes` deployments default to regular cluster pub/sub
  (RC7 behavior) until they set `pubsub-mode=sharded`.

### 2.2 `docs/2.0.0/boot3-compatibility-matrix.md`

Per-module Boot 2.7→3.x impact matrix.

**Required content:**
- **JDK baseline** — already JDK 17 (no change for 2.0.0)
- **Spring Boot** — 2.7.18 → 3.2.x (or current Boot 3 LTS at time of 2.0.0 cut)
- **Spring Framework** — 5.3.x → 6.x: javax→jakarta, autoconfigure changes
- **Per-module table** with columns: module name, javax imports (count), jakarta-impacted files,
  Spring API impact, dependency-version changes, expected fix effort (xs/s/m/l)
  - `netty-spring-web`
  - `netty-spring-webmvc`
  - `netty-spring-websocket`
  - `netty-spring-websocket-cluster`
  - `netty-spring-boot-autoconfigure`
  - 4 starter modules
  - 1 demo module
- **Library upgrades** — Lettuce 6.1.10 → 6.2+ (unlocks sharded pub/sub); jnats already current;
  Micrometer 1.9 → 1.10+ (**unlocks the Observation API for trace continuation deferred from RC6**);
  Jackson 2.x compatibility; Netty 4.1 compatibility
- **Per-feature 2.0.0 unlocks** — sharded pub/sub (cluster-design §⑪ note), Observation API trace
  continuation (release-notes §⑩), other deferred items
- **Estimated cycle shape** — # of RCs, key technical risks, test infrastructure changes needed

### 2.3 `docs/2.0.0/1.9.0-to-2.0.0-migration-guide.md` (DRAFT)

User-facing migration guide (marked DRAFT — feedback invited).

**Required sections:**
- **Audience** — existing 1.9.0 users planning Boot 3.x upgrade
- **Breaking changes (expected)** — `javax.*` → `jakarta.*` for users with custom WebSocket handlers/
  resolvers that reference Servlet API (if any); Spring API changes that affect public surface
- **Non-breaking changes** — additive features (sharded pub/sub, Observation API)
- **Step-by-step upgrade procedure** — bump dependency version, update JDK if pre-17, mass-rename
  `javax.servlet` → `jakarta.servlet` if affected, run `mvn dependency:tree` to verify, run tests
- **New feature opt-ins** — sharded pub/sub setup, Observation trace continuation
- **Config compatibility** — promise that 1.9.0 `application.yml` continues to work in 2.0.0 unless
  explicitly called out
- **DRAFT marker** — top + bottom; "this guide is a planning artifact; final guide ships with 2.0.0"
